Developing Sample API Content

May. 11, 2021

Incorporating SME API notes for documentation

In this sample API exercise, the notes below from an engineer were guidelines for building new content. This new Pet feature needed documentation to show how to use it and act as reference for parameters and expected responses.

Develop new documentation for API

To start, I evaluated the scope of the new documentation. I looked at the organization’s current API docs to base the new content off of this framework and template. This approach served as both a content audit as well as identifying the appropriate persona for the user.

Audience assessment

Comparing established documentation to this new content confirmed the audience to consist of developers and their respective background knowledge and experience. I created a functional POST example for a developer to take advantage of and use directly. This example uses valid inputs and an external link to serve as an accurate use case.

curl --location --request POST `http://api.nylas.com/pet` \
--header 'Authorization: Bearer <access_token>' \
--header 'Content-Type: application/json' \
--data-raw '{
    "category": {
    "name": "Rescue",
    "sub": {
      "prop1": "local"
    }
  },
  "name": "Guru",
  "photoUrls": [
    "https://raw.githubusercontent.com/jigsawpieces/dog-api-images/7f8ca46438aacf6aa6ed23a8c421368180370da1/husky/n02110185_9712.jpg"
  ],
  "tags": [
    {
      "name": "husky"
    }
  ],
  "status": "pending_adoption",
  "petType": "dog",
  "scopes": {
    "write:pets": "modify pets in your account",
    "read:pets": "read your pets"
  }
}'

API build

For building the new documentation, I included required elements for attributes, types, and each respective description in a table. In addition, I added a separate table for possible server responses and explanations.

Using the suggested reference specifications, I catered this new content to be faithful to the already existing documentation using Markdown and sharing it in GitHub. I maintained simple and straightforward formatting and text and included all elements of the necessary information for allowing a developer to complete the task.

Deliverable

The Lead Technical Writer provided positive feedback explaining this sample would work live as-is. Also, the notes depicted a faithful representation of the material developers share for writers to work on. This further emphasizes the quality of the sample coming from real world scenarios shaping the effectiveness of the process.
